# Finance RAG Assistant 面试讲解指南

## 1 分钟版本

我做的是一个面向财报、公告和研报的金融文档 RAG 助手。项目要解决的问题是：投研资料篇幅长、信息分散，分析师查找收入增长原因、毛利率变化和风险因素比较耗时。系统用 PyPDF 按页解析文档，用 LangChain 切分文本，本地 Embedding 后存入 Chroma；提问时先做语义检索，再把 Top K 证据交给 DeepSeek。回答会展示文件名、页码、相关度和原文片段，方便回到原始公告复核。我还在 Prompt 中约束模型区分财务事实、管理层陈述和未来计划，证据不足就拒答。因此它体现的不只是 API 调用，而是一个有证据链的 AI 辅助研究流程。

## 3 分钟版本

### 1. 业务问题

传统关键词搜索对“为什么增长”“风险来自哪里”这类语义问题不友好。整份财报直接进入大模型又会增加上下文成本，而且回答难以追溯。因此我把流程拆成文档索引和问答两个阶段。

### 2. 技术流程

索引阶段按页提取文本，保留文件名和页码 metadata；然后用中文标点和换行优先的递归切分策略生成有重叠的片段。本地多语言模型负责 Embedding，结果写入 Chroma。相同文档会通过内容指纹复用向量集合。

问答阶段用问题向量检索 Top K 片段，将每段编号为来源 1、来源 2，再通过金融场景 Prompt 让 DeepSeek回答。前端同步显示来源页码和原文，用户可以人工复核。

### 3. 金融场景设计

我没有把它做成通用问答。Prompt 特别要求保留金额单位和报告期，区分“管理层计划”与“已经实现的结果”，对证据冲突或不足明确拒答，且不输出投资建议。示例文档也覆盖收入驱动、利润率、应收账款和客户集中度等投研问题。

### 4. 局限与下一步

当前 PyPDF 对扫描件和复杂表格能力有限；只做稠密向量召回，对精确数字和生僻会计科目不一定最优。下一步会加入 OCR、BM25 + 向量混合检索、Reranker，以及包含标准证据页的评测集，用 Hit@K、MRR、引用准确率和拒答率做量化评价。

## 高频追问与参考答案

### Q1：RAG 和直接把整份 PDF 发给大模型有什么区别？

RAG 先从文档中选择与问题最相关的少量证据，再生成答案。它能降低上下文长度与调用成本，也能把答案绑定到具体来源。代价是系统多了一个检索环节，若召回失败，即使模型能力很强也无法正确回答。

### Q2：为什么 chunk size 设为 800、overlap 设为 120？

这是面向中文财务段落的初始工程参数：800 个字符通常能容纳一个完整经营分析段落，120 个字符重叠可以减少切分边界丢失因果关系。它不是普适最优值。更严谨的做法是准备标准问题集，对不同切分参数比较证据 Hit@K 和最终引用准确率。

### Q3：为什么选 Chroma 而不是 FAISS？

两者都适合本地 Demo。Chroma 自带持久化和 metadata 管理，展示文件名、页码以及后续做文档过滤更方便；FAISS 更偏高效向量索引，但 metadata 和持久化通常要额外管理。这个项目规模下性能不是主要瓶颈，因此选择开发体验更好的 Chroma。

### Q4：为什么 DeepSeek 只做生成，Embedding 用本地模型？

生成模型与向量模型职责不同。本地 Embedding 可以减少财务文档发送到外部服务、避免重复索引产生 API 成本，也让 LLM 供应商可替换。代价是首次下载模型、CPU 索引速度较慢，并且需要单独评估中文金融语义效果。

### Q5：如何保证答案没有幻觉？

不能声称完全保证。我做了四层降低风险的设计：只把检索证据交给模型；Prompt 限制文档外推断；强制关键结论附来源编号；前端展示原文供人工复核。进一步可加入引用支持度校验、低相关度拒答、答案一致性检查和人工审核。

### Q6：相关度 85% 是准确率吗？

不是。界面上的相关度由余弦距离转换而来，只表示问题和片段在当前 Embedding 空间中的接近程度，不是答案正确概率。生产系统应在验证集上校准阈值，避免把它解释为置信概率。

### Q7：如果 PDF 是扫描件或包含复杂表格怎么办？

当前 PyPDF 对可搜索文本 PDF 效果更好。扫描件需要 OCR；复杂表格应引入版面分析和表格结构识别，并把表头、行列关系一同保存。财务数字属于高风险信息，即使抽取成功也要保留页码并支持人工复核。

### Q8：如何评估这个 RAG 系统？

我会构建一组金融问题，每题标注标准证据页和参考答案。检索层用 Hit@K、MRR；生成层看答案正确性、引用准确率、引用完整率和无答案问题的拒答率；工程层记录索引耗时、单次问答延迟和调用成本。这样可以定位问题出在召回还是生成。

### Q9：如何支持多家公司和跨年度比较？

在 Document metadata 中增加证券代码、公司名、报告期和文档类型，查询前做 metadata filter。跨期问题可以先识别所需期间，再分别检索并把结果放入结构化对比模板。对核心财务指标，最好额外抽取到结构化表中，不完全依赖非结构化 RAG。

### Q10：项目最难的部分是什么？

难点不是调用 DeepSeek，而是让“回答—证据”关系可靠：页码必须在解析和切分后持续保留，Prompt 中的来源编号必须和界面片段一致，还要诚实处理证据不足。这个取舍体现了金融研究对可解释性和审计线索的要求。

### Q11：这和量化投研有什么关系？

它更偏非结构化数据工程和基本面投研，不是交易策略。它可以作为量化流程的上游：从公告中提取风险事件、业绩原因或管理层指引，再经过人工校验或结构化后形成特征。若要称为量化项目，还需要时间对齐、避免未来信息、因子构建和回测验证。

### Q12：真实上线还缺什么？

缺少用户与公司级数据隔离、权限控制、加密和脱敏、文档版本管理、任务队列、监控告警、评测流水线以及合规审计。当前版本刻意控制在本科生能够完整解释和本地复现的范围内。

- 采用内容指纹复用向量索引，设计可扩展的 metadata 与评测方案，支持后续混合检索、跨期对比和引用准确率评估。
