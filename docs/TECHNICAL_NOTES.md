# Finance RAG Assistant 技术说明与常见问题

## 项目概述

Finance RAG Assistant 是一个面向财报、公告和研报的本地金融文档 RAG 应用。它将 PDF 处理为可检索的文本片段，在用户提问时检索 Top K 证据，再由 DeepSeek 基于这些证据生成回答。Streamlit 页面同时展示文件名、页码、相关度和原文片段，便于回到原始资料复核。

项目关注非结构化金融文档处理中的几个基础问题：输入质量、文本切分、重复内容、来源追踪、向量复用、可选结构化记录以及外部依赖失败时的降级。当前实现适合本地学习、演示和技术验证，不是生产级投研系统，也不构成投资建议。

## 系统设计

系统由索引阶段和问答阶段组成。

### 索引阶段

1. 校验上传文件的扩展名、MIME、PDF 文件头、空文件、单文件大小和批次总大小。
2. 对原始文件计算 SHA-256；同一批次内内容相同的文件只保留一份。
3. 使用 PyPDF 逐页提取文本，清理空字符、多余空白和空行，并保留文件名、页码和文件哈希。
4. 使用 `RecursiveCharacterTextSplitter` 按中文标点和换行递归切分，默认 `chunk_size=800`、`chunk_overlap=120`。
5. 按“文件哈希 + 规范化片段正文”去除同一文档中的重复片段。
6. 使用本地多语言 Embedding 模型生成向量，将文本片段和元数据写入持久化 Chroma 集合。
7. 如果启用 MySQL，记录文档哈希、片段数和 `processing`、`completed` 或 `failed` 处理状态。

### 问答阶段

1. 每个问题都重新执行向量相似度检索，返回 Top K 文本片段。
2. 系统将检索结果编号为 `[来源1]`、`[来源2]` 等，并连同文件名、页码和正文发送给 DeepSeek。
3. Prompt 要求模型区分客观财务数据、管理层陈述、风险提示和未来计划，并在证据不足时说明无法确定。
4. 页面展示回答及本轮检索来源，用户可以展开查看原文片段。
5. 如果 MySQL 可用，问答记录会关联到本轮检索结果涉及的文档。该关联表示文档进入了本轮检索上下文，不等同于模型一定在最终答案中引用了该文档。

系统保留最近三轮对话语境，每轮仍独立检索当前问题的证据。历史消息用于生成上下文，不替代本轮检索。

## 输入质量控制

输入校验位于 `src/pipeline.py`，用于在 PDF 解析前尽早拒绝明显无效的输入。

- **扩展名**：仅接受 `.pdf`，比较时不区分大小写。
- **MIME**：上传对象提供 MIME 时，仅接受 `application/pdf` 和 `application/x-pdf`。
- **文件头**：原始字节必须以 `%PDF-` 开头，避免仅修改扩展名绕过类型检查。
- **空文件**：字节数为零时直接拒绝。
- **大小限制**：`MAX_UPLOAD_MB` 同时约束单个文件和单批文件总大小。
- **解析结果**：损坏、加密且无法解密、没有可提取文本的 PDF 会返回明确错误；单页提取失败时跳过该页并记录日志。

这些检查能够覆盖常见错误输入，但不能证明 PDF 内容安全，也没有实现恶意文件检测、内容合规检查或文件级反病毒扫描。

## 文本清洗、切分与重复处理

文本清洗会移除空字符和不间断空格，压缩行内多余空白并删除空行，同时保留换行结构，以减少对财务段落和简单表格阅读顺序的破坏。

切分器优先使用双换行、换行、中文句号、分号和逗号等边界。默认 800 字符通常可以容纳一个经营分析段落，120 字符重叠用于减少切分边界处的因果关系或指标口径丢失。它们只是当前工程默认值，可通过环境变量调整，不代表经过正式评测后的全局最优参数。

片段去重以文件哈希和规范化正文为联合键。相同文档内正文完全一致的片段只保留一份，不同文件中的相同文本仍分别保留来源，避免丢失文档级血缘。

## 数据血缘与向量复用

原始 PDF 的 SHA-256 在上传校验阶段生成，并沿处理链路传递：

```text
原始 PDF → ValidatedUpload.file_hash → 页级 Document metadata
         → 切分后片段 metadata → Chroma 检索结果 → MySQL 文档关联
```

LangChain 切分时会继承页级 metadata，因此文件哈希、文件名和页码能够继续保存在片段与检索结果中。页面来源展示使用文件名和页码；MySQL 使用文件哈希唯一识别文档。

Chroma 集合名称来自文档集合的稳定指纹。指纹按文件哈希、页码和正文排序计算，因此相同内容的文档集合即使上传顺序不同，也会定位到同一个集合；如果集合已有片段，系统直接复用，避免再次向量化和写入。增加、删除或修改任一文档会产生新的集合指纹。

这是一种本地缓存和基础血缘设计，不是完整的数据治理系统。当前没有文档版本关系、数据目录、血缘图、跨用户隔离、保留策略或统一数据质量规则管理。

## Chroma 与 MySQL 的职责

### Chroma

Chroma 保存文本片段、Embedding 向量以及文件名、页码、文件哈希等检索元数据，负责余弦空间中的语义相似度检索。它是当前 RAG 问答的必要组件。

### MySQL

MySQL 是可选的结构化记录层，包含：

- `documents`：文件名、SHA-256、上传时间、去重后片段数和处理状态；
- `qa_history`：文档 ID、问题、答案和创建时间。

MySQL 当前只保存文档级元数据和问答结果，不保存所有 Chroma 片段的结构化明细，也不参与向量相似度计算。

### 两者的关系

Chroma 回答“哪些文本片段与问题语义相关”，MySQL 回答“哪些文档被处理、处理结果如何、产生过哪些问答记录”。两者职责互补，不是替代关系。关闭 MySQL 不会让 Chroma 获得结构化审计能力，关闭 Chroma 则无法执行当前语义检索流程。

## 降级设计与异常处理

MySQL 默认可以关闭。以下情况会切换为不记录数据库的模式：

- `MYSQL_ENABLED=false`；
- MySQL Connector 未安装；
- 用户名或数据库名等配置不完整；
- 数据库名不符合允许的标识符格式；
- 连接、建库、建表或写入失败。

降级目标是保留 PDF 处理、Chroma 检索和 DeepSeek 问答这条核心链路。数据库失败不会被当作成功：仓储状态会标记为不可用，侧栏显示降级状态，日志记录失败原因，后续数据库写入会被跳过。恢复数据库配置后需要重启应用，使缓存的数据库仓储重新初始化。

PDF 解析、Embedding 初始化、Chroma 构建与检索、DeepSeek 调用也都有面向页面的稳定错误信息。内部异常会写入日志，避免将长堆栈或潜在敏感细节直接展示给用户。

## 可观测性

当前使用 Python 标准 `logging` 输出关键处理事件，覆盖：

- **上传**：有效文件数、重复文件数、批次大小和重复文件跳过；
- **解析**：PDF 开始与完成、有效页数、单页跳过、空文本和读取失败；
- **切分**：输入页数、生成片段数和重复片段数量；
- **Embedding**：在线模型加载失败、本地缓存回退和最终加载失败；
- **Chroma**：向量化写入、集合复用、构建失败和检索失败；
- **DeepSeek**：调用开始、完成和调用异常；空答案会转换为面向页面的友好错误；
- **MySQL**：未启用状态、初始化结果、问答关联记录和降级原因。

这些日志主要用于定位处理失败、外部依赖异常和明显的性能瓶颈。当前没有结构化日志、请求 ID、指标采集、链路追踪、集中式日志平台或告警规则，因此不能等同于生产级可观测性体系。

## 技术 FAQ

### RAG 与直接将整份 PDF 输入大模型有什么区别？

RAG 先检索与问题相关的少量文本，再生成答案。它可以减少发送给模型的上下文长度，并将页面展示绑定到具体文件、页码和原文。代价是系统增加了检索环节：如果正确证据没有被召回，生成模型无法仅凭当前上下文补救。

直接输入整份 PDF 的实现路径更短，但长文档会增加上下文成本，超出窗口时仍需裁剪，而且来源定位和重复问答成本更难控制。两种方式并非绝对优劣，取决于文档长度、问题数量、审计要求和模型上下文能力。

### `chunk_size=800` 和 `chunk_overlap=120` 如何确定？

这两个值是针对中文财务段落的初始参数。较大片段有利于保留指标、期间和原因之间的上下文，但可能引入更多无关内容；较小片段更聚焦，却容易切断句子和表格关系。Overlap 可以缓解边界损失，但会增加索引体积并产生相似片段。

合理参数应通过标注数据比较不同设置下的证据 Hit@K、MRR、引用支持度和延迟，而不是只凭经验固定。当前项目尚未完成这类参数评测。

### 为什么选择 Chroma，而不是 FAISS？

Chroma 提供本地持久化、集合管理和 metadata 存储，适合直接保存文件名、页码与文件哈希，并与 LangChain 集成。FAISS 更专注于高效向量索引，metadata、文档存储和持久化策略通常需要额外实现。

在当前小规模本地项目中，开发和来源管理便利性比极致索引性能更重要，因此选择 Chroma。这不意味着 Chroma 在所有规模或部署条件下都优于 FAISS。

### 本地 Embedding 与 DeepSeek 分别负责什么？

本地 `paraphrase-multilingual-MiniLM-L12-v2` 将文档片段和问题映射为向量，用于 Chroma 相似度检索；DeepSeek 接收问题和检索证据，负责组织自然语言答案。Embedding 与生成模型是独立组件。

本地 Embedding 可以减少文档发送到外部 Embedding 服务和重复索引 API 成本，但首次运行需要下载模型，CPU 向量化速度和金融语义效果也需要单独评估。DeepSeek 调用仍会把本轮检索到的证据发送到外部 API。

### 系统如何降低幻觉？

当前措施包括：每轮先检索证据、Prompt 要求只依据证据回答、关键判断使用来源编号、区分事实与计划、证据不足时说明无法确定，以及在页面展示原文供人工复核。

这些措施不能保证答案无幻觉。引用和拒答主要依赖 Prompt，没有程序化的引用支持度校验、低相关度阈值拒答或事实一致性验证。金融结论仍需要人工回查原始文件。

### 页面相关度是否等于答案正确率？

不等于。当前相关度由 Chroma 返回的余弦距离转换为 `1 - distance`，并截断到 0～1，仅用于直观展示当前 Embedding 空间中的接近程度。它不是统计概率，也没有经过校准，不能解释为答案置信度或正确率。

### 扫描 PDF 和复杂表格如何处理？

当前 PyPDF 适合包含可搜索文本层的 PDF。扫描件通常无法提取正文，需要先经过 OCR。复杂跨页表格可能丢失表头、行列关系或阅读顺序；当前文本清洗与切分也不会恢复完整表格结构。

更完整的方案需要 OCR、版面分析、表格结构识别以及数值口径校验。当前项目未实现这些能力。

### 如何评测 RAG 系统？

建议构建包含问题、标准证据页、参考答案和无答案问题的正式评测集，并分别评估：

- 检索层：Hit@K、MRR、证据页召回率；
- 生成层：答案正确性、引用准确率、引用完整率、拒答率；
- 工程层：索引耗时、单次检索与生成延迟、失败率和调用成本。

分层评测可以区分召回失败与生成失败。当前仓库尚未包含正式评测集或自动评测流水线，因此没有对外宣称量化提升。

### 如何扩展到多公司和多年度？

可在 Document metadata 中增加证券代码、公司名、报告期和文档类型，并在检索前进行 metadata filter。跨期问题可以先识别目标公司与期间，再分别检索并进入结构化对比流程。收入、利润率、现金流等核心指标更适合额外抽取到关系表，而不是完全依赖非结构化 RAG。

当前代码尚未实现这些 metadata 过滤、结构化指标表和跨期对比逻辑，它们属于后续扩展方向。

### 当前距离生产化还有哪些缺口？

当前仍是本地本科生项目，重点是验证完整且可解释的处理链路，而不是模拟生产系统。主要缺口包括：

- 没有 OCR、Hybrid Search、Reranker 和正式评测集；
- 没有用户、租户或公司级数据隔离和权限控制；
- 没有敏感信息识别、脱敏、静态加密和密钥托管；
- 没有异步任务队列、并发控制、限流、重试编排和成本预算；
- 没有文档版本管理、全量片段结构化明细和完整数据血缘；
- 没有指标监控、链路追踪、集中式日志、告警和合规审计；
- 没有经过真实业务数据验证的检索阈值、容量规划或服务等级目标。

这些限制决定了当前实现应被描述为本地技术项目，而不是生产级投研平台。
